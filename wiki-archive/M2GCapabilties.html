Capabilities of the Migration Tool

This section describes the capabilities of the Migration Tool for Project GlassFish.

  • Configuration Parameter Migration
  • Deployment Descriptors (DD) Migration
  • Custom JSP Tags Implementation
  • Proprietary API Implementation
  • IDE Integration
  • Build Scripts Generation
  • Logging

Configuration Parameter Migration

As a design guideline, the Migration Tool migrates only the JDBC data sources. It does not overwrite the configuration parameters of the application server installation. This design implementation protects the configuration work performed by the system administrator on the application server. Therefore, configuration parameters from source application servers have been categorized into the following migration resolutions:

  • Parameters that are Migrated to the application server.
  • Parameters that Cannot Be Migrated to the application server because there is no equivalent parameter for the application server.
  • Parameters that Should Not Be Migrated to the application server; the Migration Tool should not over-rule decisions made by the system administrator.
  • Parameters that Need Not Be Migrated to the application server because they are application server-specific (such as, WL_HOME, installation directory, and so on).
  • Parameters that are Not Migrated.

Deployment Descriptors (DD) Migration

The Migration Tool migrates the application server-specific DDs from the source application server to the Application Server.

DD Migration for WAS 4.0

The following WebSphere-specific DD files are supported:

  • ejb-jar.xml (server-independent DD, required for converting other WebSphere-specific DDs)
  • ibm-ejb-jar-bnd.xmi (contains JNDI bindings of EJB components)
  • ibm-ejb-jar-ext.xmi (server-specific information about the EJB components such as session timeout)
  • Map.mapxmi (if CMP EJB components contain mappings of CMP fields to Database columns)
  • Schema.dbxmi (if CMP EJB components contain mappings of database schema)
  • web.xml (server-independent DD, required for converting other WebSphere specific DDs)
  • ibm-web-bnd.xmi (used for binding JNDI names to the Environment Naming Context of Servlets/JSPs declared in the web.xml file
  • ibm-web-ext.xmi (contains the error message to be displayed to the user in case a component of the web application fails.)
    Relevant information from the above DD files is used to generate the following Application Server-specific DDs:
  • ejb-jar.xml (remains the same for the Application Server)
  • sun-ejb-jar.xml
  • sun-cmp-mappings.xml (contains information about CMP entity beans; its name is specified in sun-ejb-jar.xml)
  • web.xml (remains the same for the Application Server)
  • sun-web.xml

DD Migration for WLS 5.1

The following WebLogic specific DD files are supported:

  • ejb-jar.xml (server-independent DD, required for converting other WebLogic-specific DDs)
  • weblogic-ejb-jar.xml (mapping of the properties declared in the ejb-jar.xml to the server specific properties)
  • weblogic-rdbms-persistence.xml (contains information about CMP entity beans; its name is specified in weblogic-ejb-jar.xml)
  • web.xml (server-independent DD, required for converting to other WebLogic-specific DDs)
  • weblogic.xml (mapping to the server-specific of properties declared in the web.xml DD)
    Relevant information from the above DD files is used to generate the following Application Server-specific DDs:
  • ejb-jar.xml (remains the same for the application server)
  • sun-ejb-jar.xml
  • sun-cmp-mappings.xml (contains information about CMP entity beans; its name is specified in sun-ejb-jar.xml)
  • web.xml (remains the same for the application server)
  • sun-web.xml

Build Scripts Generation

The Migration Tool automatically generates Ant-based build scripts that can be used to compile and package the migrated application. Ant is an open source Java project used by many companies to simplify cross-platform software development. The Migration Tool generates three types of scripts for Solaris and Windows:

  • build_jar.sh / build_jar.cmd
  • build_war.sh / build_war.cmd
  • build_ear.sh / build_ear.cmd
    These helper scripts run Ant on the corresponding XML (build_jar.xml, build_war.xml, or build_ear.xml) build script.
    These Ant scripts are created with the following assumptions:
  • The build_jar script, when executed, creates a JAR file that contains the enterprise beans and other resources (helper classes and so on) packaged and ready for deployment. The script executes the following steps:

Using javac, the script compiles all classes that are part of the JAR file.
Using the Ant jar task, the script creates the JAR file, which contains all the class files and the XML DDs that are a part of the enterprise beans and helper classes used by the enterprise beans.

  • The build_war script, when executed, creates a WAR file that contains the servlets, JSPs, and other resources (helper classes, images, and so forth that were part of the input to the Migration Tool), packaged and ready for deployment. To do this, the script executes the following steps:

Using javac, the script compiles all classes that are part of the WAR file.
Using the Ant war task, the script creates the WAR file, which contains all the Web application resources (servlet classes, JSPs, images, helper classes, and so forth) and the XML DDs that are part of the Web application.

  • The build_ear script, when executed, creates an EAR file that contains the JAR and WAR files (previously created in steps 1 and 2), packaged and ready for deployment. To do this, the script uses the Ant ear task to create the EAR file, which contains the WAR and JAR files, along with the XML DDs that are part of the enterprise application.

Bundling of Class Files

If the input application contains .class files only, then the generated build scripts do not do any compilation, they only package the class files back with the migrated DDs.
If the input application contains Java source files along with class files, the generated build scripts invoke the javac Ant target and package only the compiled classes, not the orignal classes. The Limitations section discusses this requirement in greater detail.

Logging

The Migration Tool generates debug and user logging information. These logs, found in the <INSTALL_DIR>\asmt\logs}}directory, are called {{ASMTDebug.log and ASMTUser.log.
The ASMTDebug.log file contains debugging information generated during migration. A developer may find this information useful for tracking the migration flow, since the log is designed to enable tracing of errors and other unexpected behavior that may result in bugs. The log contains migration implementation details indicating how things were done during migration. Although information in this log may not be understood by users, it is imperative that users provide this log when reporting bugs or other unexpected behavior.
The ASMTUser.log is a heavily-filtered form of the ASMTDebug.log file. This log, which contains information specific to areas of migration that interest the user, indicates what things were done during migration. For example, when migrating deployment descriptors, the Migration Tool may find that certain deployment descriptors do not adhere to their DTDs. Since in these cases the Migration Tool modifies the deployment descriptors as required, it uses the ASMTUser.log file to notify the user about the modified files. In addition to notifications, this log contains information about the generation of build scripts, automatic generation of the build process, brief descriptions of exceptions encountered, and so forth. You need to provide this log when you report bugs or other unexpected behavior.
