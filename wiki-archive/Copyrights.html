Copyright and License HOWTO

This page describes how to create, update, and maintain the copyright and license notice that should appear in every GlassFish source file.

General Rules

What to copyright

Wherever possible, text files should contain a comment that includes a copyright notice and a license notice. The comment should use the syntax appropriate for the type of file. For example, .java files will use Java syntax comments, XML files will use XML syntax comments, and so on. See below for examples in different formats.

Copyright years

The copyright notice should include the year the file was created. When a file is updated, the copyright notice should be updated to include the current year as well. Only the first year and last year should be included. A range of years is expressed as two years separated by a dash, with no spaces, e.g., "1997-2010".

Comment format

In general, the comment containing the copyright and license notice should be the first text in the file and should be followed by a blank line. The following exceptions are allowed:

  • A shell script may start with a "#!" line.
  • A Python file may have a "# -*- coding:" line after the "#!" line.
  • An XML format file may start with a "<?xml>" line, and may have a "<!DOCTYPE>" line before the comment.
  • An html file may have "<html>" and "<head>" lines before the comment.

The Common Case

Source files created by Oracle employees will typically use one of two licenses:

  1. Most source code will use the CDDL and GPL license with the Classpath Exception.
  2. Demo or sample source code should use the BSD license.

The copyright/license header should be the first text in the file, with exceptions noted above.

Third Party Files

Files that are imported from a third party, such as Apache licensed files, are handled differently. (Remember that you always need approval before importing any third-party code, no matter what the license is.) If the file is imported with no changes, or with only minor changes, the original copyright and license must be maintained.

The following general rules apply to third-party files:

  • Always obtain approval before adding any third-party files to the workspace.
  • Never alter or delete a third-party copyright notice, unless explicitly instructed by an Oracle attorney.
  • Never delete other text that appears to claim ownership, unless explicitly instructed by an Oracle attorney. Examples include phrases such as "Property of ..." or "Created by ...".
  • If a third-party file does not contain a third-party copyright notice, do not insert such a notice.
  • If a file contains a third-party copyright notice but no license notice or other statement of permission, and if the license text is in a separate file near the source file, retain the separate file but do not insert that license text into the source file.

When the amount of change in a third-party file is significant, we add the Oracle copyright and license notice to the file while preserving the original copyright and license notice. See below for an example using the Apache license. Note that this approach only works for third-party code that uses a liberal license that allows us to relicense the code under our license. Examples of such licenses are the Apache license, the BSD license, and the MIT license. GPL and similar licenses do not allow this approach. When in doubt, ask for help.

How do you tell when the amount of change is significant? The following guidance from the U.S. Copyright office may be helpful if you need to determine whether modifications to a file have transformed that file into a derivative work. To be copyrightable, a derivative work must be different enough from the original to be regarded as a new work or must contain a substantial amount of new material. Making minor changes or additions of little substance to a preexisting work will not qualify the work as a new version for copyright purposes. The new material must be original and copyrightable in itself. Titles, short phrases, and format, for example, are not copyrightable.

As always, if in doubt then ask for help.

Other Cases

For any other cases, please contact your manager, your product manager, or Bill Shannon for assistance.

Portions Copyright

If you read the license notice carefully, you'll see that it has instructions for contributors who want to copyright their modifications to a file. In such a case, a separate comment block should be added after the main copyright/license comment block with this additional information in the following format:

/*
 * Portions Copyright [year] [name of copyright owner]
 */

Updating A File

When updating a file, verify that the license header meets the above requirements.
Also, ensure that the copyright includes the current year as well as the first year that the file was created; it should be either a single year (e.g., "2010"), or a range of years separated by a dash with no spaces (e.g., "1997-2010").

Generated Source Files

Files generated at build time and shipped only in binary form (including minimized .js files) do not need a license or copyright header.

Files generated at build time and shipped in source form do need a copyright and license header.

Files generated at run time based on some user code do not need a copyright or license header. (For example, .java files generated for JSP pages, deployment descriptors generated for annotations.)

Files generated at run time without depending on any user code do need a copyright and license header. (For example, start/stop scripts, asenv.conf file, domain.xml.)

Maven Copyright Plugin

You can check that your copyright notices are correct using the Maven copyright plugin in projects
that have it configured in the pom.xml:

$ mvn glassfish-copyright:copyright

To repair any errors discovered (use this carefully, and check the results):

$ mvn glassfish-copyright:repair

The copyright plugin is configured to use a common exclude list in the maven artifact org.glassfish:copyright. If you get complaints about this, you'll need to build this artifact explicitly (it's not built by a normal full build). Use either:

$ mvn -Prelease-phase1 install

or

$ cd copyright; mvn install

For more information, see the README file at http://java.net/projects/glassfish/sources/svn/show/trunk/copyright-plugin.

For projects that don't (yet) have the plugin configured in their pom.xml, you can run it using:

mvn org.glassfish.copyright:glassfish-copyright-maven-plugin:copyright -Pcopyright
